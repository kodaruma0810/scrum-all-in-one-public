## 1. このアプリでできること

### 全体像

```
組織(Organization)
  └── チーム(Team)
        ├── インクリメント(PI) ← 数ヶ月単位の計画期間
        │     ├── 長期ゴール ← PI期間中に達成したい目標
        │     └── スプリント ← 1〜2週間の作業サイクル
        │           ├── スプリントゴール ← スプリントで達成したい目標
        │           └── チケット ← 実際の作業単位
        ├── デイリースクラム(DSU)
        ├── レトロスペクティブ
        └── ワーキングアグリーメント
```

### 機能一覧

| カテゴリ | できること |
|----------|-----------|
| **認証** | ユーザー登録、ログイン、APIキー発行 |
| **チケット管理** | ユーザーストーリー/タスク/バグの作成・管理、カンバンボード、ドラッグ&ドロップ |
| **ゴール管理** | インクリメント(PI)の作成、長期ゴール・スプリントゴールの階層管理 |
| **スプリント管理** | スプリントの計画・開始・完了、キャパシティ管理、ベロシティ追跡 |
| **デイリースクラム** | 出欠管理、Yesterday/Today/Blockers の記録、タイマー、バーンダウンチャート |
| **レトロスペクティブ** | KPT形式の振り返り、カード/ホワイトボード表示、投票、アクションアイテム管理 |
| **ワーキングアグリーメント** | チームルールの管理、カテゴリ分類、変更履歴、公開共有リンク |
| **ダッシュボード** | スプリント進捗、ゴール達成状況、自分のチケット、ベロシティ推移 |
| **検索** | チケット・ゴールの横断検索（Ctrl+K でいつでも呼び出せる） |
| **ユーザー管理** | ロール管理（ADMIN/MEMBER）、プロフィール編集 |
| **チーム管理** | チーム作成、メンバー追加、チームロール設定 |
| **用語カスタマイズ** | 「スプリント」「インクリメント」などの表示名を自由に変更可能 |
| **モバイル対応** | スマホ・タブレットからも使える |

## 3. 起動・停止のやり方

### 開発環境（ローカル開発）

コードを書き換えながら開発する場合の手順です。ファイルを保存すると自動で反映されます。

#### 初回セットアップ

```bash
# 1. プロジェクトのフォルダに移動
cd scrum-all-in-one

# 2. 依存パッケージのインストール
pnpm install

# 3. 環境変数ファイルを作成（初回のみ）
cp .env.example .env.local

# 4. ローカルのデータベースを起動（Supabase）
pnpm supabase:start

# 5. Prisma クライアントを生成
pnpm db:generate

# 6. データベースのマイグレーション
pnpm db:migrate

# 7. 初期データの投入（初回のみ）
pnpm db:seed

# 8. 開発サーバーを起動
pnpm dev
```

#### 2回目以降の起動

```bash
# 1. Docker Desktop を起動（起動していなければ）
# 2. Supabase を起動
pnpm supabase:start

# 3. 開発サーバーを起動
pnpm dev
```

#### 停止

```bash
# Ctrl+C で開発サーバーを停止した後
pnpm supabase:stop
```

#### 開発サーバーのアクセス先

- フロントエンド: `http://localhost:5173`
- バックエンド API: `http://localhost:4000`
- **Supabase Studio（DB管理画面）: `http://127.0.0.1:54323`**

> Supabase Studio でテーブルの中身をブラウザから閲覧・編集できます。

#### データベースをリセットしたい場合

```bash
pnpm supabase:reset
```

> マイグレーションが再実行され、シードデータも再投入されます。

#### 便利コマンド

| コマンド | 何をする？ |
|---------|-----------|
| `pnpm dev` | フロントエンド＆バックエンドを同時起動 |
| `pnpm dev:frontend` | フロントエンドだけ起動 |
| `pnpm dev:backend` | バックエンドだけ起動 |
| `pnpm supabase:start` | ローカルDB（Supabase）を起動 |
| `pnpm supabase:stop` | ローカルDBを停止 |
| `pnpm supabase:reset` | DBをリセット（データ全削除＋再作成） |
| `pnpm supabase:status` | Supabase の起動状態を確認 |
| `pnpm db:studio` | Prisma Studio を起動（もう一つのDB管理画面） |
| `pnpm build` | 本番用にビルド |

### セルフホスト（本番デプロイ）

Docker Compose を使って本番環境にデプロイする場合の手順です。
開発には使いません。

#### 初回起動（セットアップ含む）

```bash
# 1. プロジェクトのフォルダに移動
cd scrum-all-in-one

# 2. セットアップスクリプトを実行（初回のみ）
bash setup.sh
```

`setup.sh` が自動でやってくれること：
- 設定ファイル（`.env.production`）の作成
- パスワードやシークレットキーの自動生成
- Docker コンテナのビルドと起動
- データベースの初期データ投入

#### 2回目以降の起動

```bash
docker compose up -d
```

#### 停止

```bash
docker compose down
```

#### コードを変更した後の再起動

```bash
docker compose up -d --build
```

> `--build` をつけると変更を反映してイメージを作り直します。

#### データも含めて全部リセットしたい場合

```bash
docker compose down -v
```

> `-v` でデータベースやRedisのデータも削除されます。再起動後は初期状態に戻ります。

#### Docker の便利コマンド

| コマンド | 何をする？ |
|---------|-----------|
| `docker compose ps` | 動いているコンテナの一覧を見る |
| `docker compose logs -f` | 全サービスのログをリアルタイムで見る |
| `docker compose logs -f backend` | バックエンドだけのログを見る |
| `docker compose restart backend` | バックエンドだけ再起動する |

#### セルフホスト時のサービス構成

```
ブラウザ
  ↓ http://localhost (ポート80)
[Nginx リバースプロキシ]
  ├── /api/* → Backend (ポート4000)
  └── それ以外 → Frontend (ポート3000)

[PostgreSQL] ← データベース (ポート5432)
[Redis]      ← キャッシュ (ポート6379)
```

> ブラウザからは `http://localhost` でアクセスできます。

## 4. 初回セットアップ

### デフォルトのログイン情報

初回起動後、以下のアカウントが自動で作成されます：

| 項目 | 値 |
|------|-----|
| メールアドレス | `admin@example.com` |
| パスワード | `admin123` |

> **重要**: 本番環境ではすぐにパスワードを変更してください。

### 最初にやること

1. **ログイン** → デフォルトアカウントでログイン
2. **チームを作成** → サイドバーのチーム管理からチームを作成
3. **メンバーを招待** → 他のユーザーをチームに追加
4. **インクリメント(PI)を作成** → ゴール管理の起点を作る
5. **スプリントを作成** → 作業サイクルを設定
6. **チケットを登録** → バックログに作業を追加

## 5. 画面ごとの使い方

### 5.1 ダッシュボード (`/dashboard`)

ログイン後のホーム画面です。以下のウィジェットを表示できます：

| ウィジェット | 表示内容 |
|-------------|---------|
| スプリントサマリー | アクティブなスプリントの進捗（残日数、完了率） |
| 自分のチケット | 自分に割り当てられたチケットの一覧 |
| ゴール進捗 | 長期ゴールの達成状況 |
| ベロシティ推移 | 過去スプリントのベロシティグラフ |
| アクションアイテム | レトロで出た未完了のアクションアイテム |
| ワーキングアグリーメント | チームルールの確認 |

> ウィジェットは自分で選んでカスタマイズできます。設定はブラウザに保存されます。

### 5.2 バックログ (`/backlog`)

すべてのチケットを一覧で管理する画面です。

- チケットの新規作成
- スプリントへの割り当て
- フィルター（ステータス、担当者、タイプ、優先度）
- チケットのドラッグ&ドロップによる並び替え

### 5.3 インクリメント / PI (`/pi`)

数ヶ月単位の計画期間を管理します。

- PI の作成・編集・削除
- 長期ゴールの設定（優先度: 必須 / あった方がいい / できれば）
- スプリントゴールとの紐付け
- ゴール階層ツリー / マトリクスビューで進捗を俯瞰

### 5.4 スプリントプランニング (`/pi/:id/planning/:sprintId`)

スプリントの計画を立てる画面です。

- バックログからチケットをスプリントに移動
- キャパシティの設定（メンバーごとの稼働日数）
- スプリントゴールの設定
- ストーリーポイントの見積もり

### 5.5 アクティブスプリント (`/sprint`)

実行中のスプリントを管理する画面です。以下の機能がまとまっています：

**カンバンボード**
- ステータス列: BACKLOG → TODO → IN_PROGRESS → IN_REVIEW → DONE
- チケットをドラッグ&ドロップでステータス変更

**デイリースクラム (DSU)**
- メンバーの出欠（出席 / 欠席 / リモート）
- Yesterday（昨日やったこと）/ Today（今日やること）/ Blockers（困っていること）
- バーンダウンチャート
- カウントダウンタイマー（15分デフォルト）
- スプリントゴール別の進捗表示

### 5.6 チケット詳細 (`/tickets/:id`)

個別のチケットを閲覧・編集する画面です。

- 基本情報の編集（タイトル、説明、タイプ、優先度、ストーリーポイント）
- 担当者の変更
- サブタスクの追加
- コメントの投稿
- DoD（完了の定義）チェックリスト
- ステータス変更

**チケットタイプ**

| タイプ | 用途 |
|--------|------|
| USER_STORY | ユーザーに価値を届ける機能単位 |
| TASK | 技術的な作業 |
| BUG | 不具合の修正 |
| SUBTASK | 上記の子タスク |

**優先度**

HIGHEST > HIGH > MEDIUM > LOW > LOWEST

### 5.7 レトロスペクティブ (`/retrospectives`)

スプリントの振り返りを行う画面です。

- **KPT形式**: Keep（続けたいこと）/ Problem（問題）/ Try（試したいこと）
- **カードモード**: カードを一覧で表示
- **ホワイトボードモード**: 自由に配置できるボード
- 各カードに投票できる
- アクションアイテムを作成して担当者を割り当て
- アクションアイテムのステータス管理（OPEN / DONE）

### 5.8 ワーキングアグリーメント (`/working-agreements`)

チームのルール・約束事を管理する画面です。

- カテゴリ別にルールを整理
- ルールの有効/無効の切り替え
- 変更履歴の自動記録
- **共有リンク**: 外部に公開できる読み取り専用ページ

### 5.9 検索 (`/search`)

チケット・長期ゴール・スプリントゴールを横断検索します。

- **ショートカット**: `Ctrl+K`（Mac: `Cmd+K`）でいつでも呼び出せる
- タイトル・説明文をまとめて検索

### 5.10 プロフィール (`/profile`)

自分のプロフィール（名前、アバター等）を編集する画面です。

### 5.11 APIキー管理 (`/api-keys`)

外部ツールやスクリプトからAPIにアクセスするためのキーを管理します。

- APIキーの発行（`sk_` プレフィックス付き）
- 有効期限の設定
- キーの失効（取り消し）
- 使用方法: リクエストヘッダーに `Authorization: ApiKey sk_xxxxx` を付ける

## 6. 管理者向けの設定

### 6.1 ユーザー管理 (`/users`)

ADMIN ロールのユーザーのみアクセスできます。

- ユーザーの一覧表示
- 新規ユーザーの作成
- ユーザーロールの変更（ADMIN / MEMBER）
- ユーザーの削除

### 6.2 チーム管理 (`/team-management`)

- チームの作成・削除
- チームメンバーの追加・削除
- メンバーのチームロール変更

**チームロール**

| ロール | 主な権限 |
|--------|---------|
| SCRUM_MASTER | スプリント管理、DSU管理 |
| PRODUCT_OWNER | ゴール管理、バックログ管理 |
| DEVELOPER | チケット操作、DSU入力 |

### 6.3 チーム設定

| 設定項目 | 説明 |
|---------|------|
| チケットプレフィックス | チケット番号の接頭辞（例: `SCR-1`, `PRJ-1`） |
| ベロシティモード | ストーリーポイント or チケット数 で計測 |
| SP→日数換算レート | 1SPが何日分かの目安 |

### 6.4 用語カスタマイズ

「インクリメント」「スプリント」などの用語を、チームに合った呼び方に変更できます。

| デフォルト用語 | カスタマイズ例 |
|---------------|-------------|
| インクリメント | PI / フェーズ / クォーター |
| スプリント | イテレーション / サイクル |
| 長期ゴール | PIゴール / 四半期目標 |
| スプリントゴール | イテレーションゴール |

### 6.5 DoD（完了の定義）チェックリスト

チームごとに「チケットが完了するための条件」を定義できます。

- チェック項目の追加・編集・削除
- 各チケットでチェックリストとして表示
- 誰がチェックしたかの記録

### 6.6 チームカレンダー

メンバーの休暇や不在を登録できます。スプリントのキャパシティ計算に反映されます。

## 9. よくあるトラブルと対処法

### Docker が起動しない

**症状**: `docker compose up` でエラーになる

```bash
# Docker Desktop が起動しているか確認
docker info

# 起動していなければ Docker Desktop を開いてから再実行
docker compose up -d --build
```

### ポート 80 が既に使われている

**症状**: `bind: address already in use` エラー

```bash
# 何がポート80を使っているか確認（Windows）
netstat -ano | findstr :80

# 別のアプリ（Skype, IIS 等）がポート80を使っている場合は
# そのアプリを停止するか、docker-compose.yml の proxy のポートを変更
# 例: "8080:80" に変更すると http://localhost:8080 でアクセス
```

### データベースに接続できない

**症状**: バックエンドのログに `Can't reach database server` と出る

```bash
# DBコンテナの状態を確認
docker compose ps db

# DBが healthy でなければ再起動
docker compose restart db

# それでもダメなら全部リセット
docker compose down -v
docker compose up -d --build
```

### 画面が真っ白になる

**症状**: `http://localhost` にアクセスしても何も表示されない

```bash
# コンテナが全部動いているか確認
docker compose ps

# フロントエンドのログを確認
docker compose logs frontend

# Nginx のログを確認
docker compose logs proxy
```

### ログインできない

- メールアドレス: `admin@example.com`
- パスワード: `admin123`
- 上記で入れない場合は、DBをリセットして再シード：

```bash
docker compose down -v
docker compose up -d --build
# setup.sh を再実行して初期データを投入
```

### ローカル開発で Supabase が起動しない

```bash
# Supabase CLI がインストールされているか確認
supabase --version

# Docker が起動しているか確認（Supabase もDockerを使います）
docker info

# リセットして再起動
pnpm supabase:stop
pnpm supabase:start
```

### 環境変数の設定

`.env.example` をコピーして設定します：

| 変数名 | 説明 | 例 |
|--------|------|-----|
| `DATABASE_URL` | DB接続先 | `postgresql://postgres:postgres@localhost:54322/postgres` |
| `JWT_SECRET` | JWTの署名キー | ランダムな長い文字列 |
| `JWT_REFRESH_SECRET` | リフレッシュトークンの署名キー | ランダムな長い文字列 |
| `FRONTEND_URL` | フロントエンドのURL | `http://localhost:3000` |
| `PORT` | バックエンドのポート | `4000` |
| `NODE_ENV` | 環境 | `development` or `production` |
| `POSTGRES_PASSWORD` | DBパスワード（Docker用） | ランダムな文字列 |

> Docker（`setup.sh`）を使う場合は自動生成されるので、手動設定は不要です。

## 補足: APIの使い方（外部連携）

APIキーを発行すれば、外部のツールやスクリプトからデータにアクセスできます。

### APIキーの発行

1. 画面右上のメニュー → 「APIキー」
2. 「新しいキーを作成」をクリック
3. 表示されたキー（`sk_` で始まる文字列）をコピーして安全に保管

### APIの呼び出し例

```bash
# チケット一覧を取得する例
curl -H "Authorization: ApiKey sk_あなたのキー" \
     -H "X-Team-Id: チームのID" \
     http://localhost/api/tickets
```

### 主なAPIエンドポイント

| メソッド | パス | 説明 |
|---------|------|------|
| GET | `/api/tickets` | チケット一覧 |
| POST | `/api/tickets` | チケット作成 |
| GET | `/api/sprints` | スプリント一覧 |
| GET | `/api/goals/increments` | インクリメント一覧 |
| GET | `/api/dashboard/summary` | ダッシュボードデータ |
| GET | `/api/search?q=キーワード` | 検索 |

> すべてのチーム関連APIには `X-Team-Id` ヘッダーが必要です。
