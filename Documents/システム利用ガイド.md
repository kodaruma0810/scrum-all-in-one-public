## 1. システム概要

Scrum All-in-One は、中長期ゴール → スプリントゴール → チケットの3層階層を明示的に表現し、DSU・スプリント計画をすべて1つのアプリで完結できる Scrum 管理ツールです。

### 主な機能

| 機能 | 概要 |
|---|---|
| チケット管理 | カンバンボード、バックログ、バーンダウンチャート、DoDチェックリスト |
| ゴール管理 | インクリメント（PI）単位の中長期ゴール、スプリントゴール、階層ツリー/マトリクスビュー |
| スプリント管理 | スプリント作成・開始・終了、プランニングボード、キャパシティ管理、チームカレンダー |
| DSU | スプリントサマリー、ゴール別進捗、メンバーステータス入力、カウントダウンタイマー、全画面モード |
| ダッシュボード | スプリント進捗、ゴール達成状況、担当チケット、ベロシティ推移 |
| グローバル検索 | チケット・ゴール横断検索（Cmd/Ctrl+K ショートカット対応） |
| ユーザー・チーム管理 | ユーザーCRUD、ロール管理、チーム設定、用語カスタマイズ |

### ユーザーロール

| ロール | 権限 |
|---|---|
| ADMIN | 全操作（ユーザー管理、チーム設定、用語カスタマイズ含む） |
| SCRUM_MASTER | スプリント管理、DSU管理、チケット管理 |
| PRODUCT_OWNER | ゴール管理、バックログ管理、チケット管理 |
| DEVELOPER | チケット操作、DSU入力、コメント投稿 |

## 2. 環境構築・起動方法

### 2.1 前提条件

- **Node.js** 18以上
- **pnpm** 8以上
- **Docker / Docker Compose**（本番環境・DB利用時）
- **Git**

### 2.2 初期セットアップ

```bash
# リポジトリのクローン
git clone <repository-url>
cd scrum-all-in-one

# 依存パッケージのインストール（Prismaクライアントも自動生成される）
pnpm install
```

### 2.3 起動方法

#### A. DB不要の開発モード（UIの確認のみ）

データベースを使わず、モックデータでフロントエンドの動作を確認できます。

```bash
# バックエンド起動（モックルーターが自動で有効化）
pnpm dev:backend

# 別ターミナルでフロントエンド起動
pnpm dev:frontend
```

- フロントエンド: http://localhost:3000
- バックエンドAPI: http://localhost:4000

ログイン画面で「**Dev Login（DB不要）**」ボタンをクリックすると、データベース接続なしで全画面を確認できます。

#### B. Docker Compose でフル環境起動

```bash
# .envファイルの準備
cp .env.example .env
# 必要に応じて JWT_SECRET 等を変更

# DB・Redis・MinIO + アプリを全て起動
docker compose up -d

# DBマイグレーション実行（初回のみ）
pnpm db:migrate

# （任意）シードデータの投入
pnpm db:seed
```

- アプリ（Nginx経由）: http://localhost
- フロントエンド直接: http://localhost:3000
- バックエンドAPI直接: http://localhost:4000
- MinIO管理画面: http://localhost:9001
- PostgreSQL: localhost:5432

#### C. DB + ローカル開発の併用

```bash
# Docker でDB・Redis・MinIOだけ起動し、アプリはローカルで実行
pnpm dev
```

このコマンドは `docker compose up -d db redis storage` を実行した後、バックエンドとフロントエンドを並列起動します。

### 2.4 環境変数

`.env` ファイルで設定します（`.env.example` を参照）。

| 変数名 | 説明 | デフォルト値 |
|---|---|---|
| `DATABASE_URL` | PostgreSQL接続文字列 | `postgresql://scrum_user:scrum_pass@localhost:5432/scrum_db` |
| `REDIS_URL` | Redis接続文字列 | `redis://localhost:6379` |
| `JWT_SECRET` | JWTアクセストークンの署名鍵 | （要設定） |
| `JWT_REFRESH_SECRET` | JWTリフレッシュトークンの署名鍵 | （要設定） |
| `MINIO_ENDPOINT` | MinIOホスト | `localhost` |
| `MINIO_PORT` | MinIOポート | `9000` |
| `MINIO_ACCESS_KEY` | MinIOアクセスキー | `minioadmin` |
| `MINIO_SECRET_KEY` | MinIOシークレットキー | `minioadmin123` |
| `MINIO_BUCKET` | ファイルアップロード用バケット名 | `scrum-uploads` |
| `FRONTEND_URL` | フロントエンドのURL（CORS設定） | `http://localhost:3000` |
| `PORT` | バックエンドのポート | `4000` |
| `NODE_ENV` | 環境モード | `development` |

> **注意**: `DATABASE_URL` が未設定の場合、バックエンドは自動的にモックルーターを有効化し、ダミーデータを返します。本番環境では必ず設定してください。

## 3. ユーザー向けガイド

### 3.1 ログイン

1. ブラウザで http://localhost:3000（またはデプロイ先URL）にアクセス
2. メールアドレスとパスワードを入力して「ログイン」をクリック
3. 認証成功後、ダッシュボードに遷移

**開発環境のみ**: 「Dev Login（DB不要）」ボタンでDB接続なしにログインできます。

### 3.2 ダッシュボード（/dashboard）

ログイン後のホーム画面です。以下のウィジェットが表示されます。

- **スプリントサマリー**: アクティブスプリントの進捗率、残日数、チケット消化状況
- **ゴール達成状況**: 中長期ゴールの進捗一覧（優先度・ステータス付き）
- **自分のチケット**: 担当チケットの一覧（直近のもの、ステータス付き）
- **ベロシティ推移**: 過去スプリントのベロシティをグラフ表示

### 3.3 チケット管理

#### チケット一覧（/tickets）

- 全チケットをカンバンボード形式で表示
- **ステータス列**: Backlog → To Do → In Progress → In Review → Done
- チケットカードをドラッグ&ドロップしてステータスを変更可能
- フィルタ: スプリント、ゴール、ステータス、担当者で絞り込み

#### チケット作成

1. チケット一覧画面の「新規作成」ボタンをクリック
2. モーダルで以下を入力:
   - **タイトル**（必須）
   - **種別**: ユーザーストーリー / タスク / バグ / サブタスク
   - **説明**: 詳細な内容
   - **優先度**: Highest / High / Medium / Low / Lowest
   - **ストーリーポイント**: フィボナッチ数列（1, 2, 3, 5, 8, 13, 21）
   - **担当者**: チームメンバーから選択
   - **スプリント**: 所属するスプリント（未割当ならバックログ）
   - **スプリントゴール**: 紐づけるゴール

#### チケット詳細（/tickets/:id）

- チケットの全情報を表示
- コメントの閲覧・投稿
- DoDチェックリストのチェック操作
- ステータス変更履歴の確認

#### バックログ（/backlog）

- スプリント未割当のチケット一覧
- 優先度順に表示
- ドラッグ&ドロップで並び替え可能

#### バーンダウンチャート

- スプリント詳細画面に表示
- 残ストーリーポイントの推移を折れ線グラフで表示
- 理想線（均等消化）をオーバーレイ表示

### 3.4 ゴール管理

#### インクリメント一覧（/goals）

- インクリメント（3ヶ月単位の期間）の一覧を表示
- 各インクリメントの中長期ゴール数と進捗率を表示

#### インクリメント詳細（/goals/:id）

- 選択したインクリメントに紐づく中長期ゴールの一覧
- ゴール作成・編集・削除
- 各ゴールの進捗バーを表示

#### 中長期ゴール属性

| 属性 | 説明 |
|---|---|
| タイトル | ゴール名称（必須） |
| 詳細説明 | リッチテキスト |
| 優先度 | Must Have / Should Have / Nice to Have |
| ステータス | 未着手 / 進行中 / 達成 / 未達成 / 一部達成 |
| コミットメント | Committed / Uncommitted |
| 担当者 | チームメンバーから選択 |

#### 階層ビュー（/goals/:id/hierarchy）

2つの表示モードを切り替えできます。

- **ツリービュー**: 中長期ゴール → スプリントゴール → チケットを木構造で展開表示。折りたたみ可能。
- **マトリクスビュー**: 行=中長期ゴール、列=スプリントのグリッド形式。ゴールとスプリントの関係を俯瞰。

### 3.5 スプリント管理

#### スプリント一覧（/sprints）

- タイムライン形式でスプリントを表示
- ステータス（計画中/アクティブ/完了）で色分け

#### スプリント作成

1. スプリント一覧の「新規作成」ボタンをクリック
2. 入力項目:
   - **名称**（例: "Sprint 2026-Q2-1"）
   - **開始日・終了日**（デフォルト2週間）
   - **所属インクリメント**
   - **目標サマリー**

#### スプリント詳細（/sprints/:id）

- スプリントゴール一覧
- チケット一覧（カンバンボード）
- キャパシティサマリー
- スプリント開始/終了ボタン

#### スプリントプランニング（/sprints/:id/planning）

- **左ペイン**: バックログのチケット一覧（ドラッグ元）
- **右ペイン**: スプリントゴール + チケット割当（ドロップ先）
- **上部バー**: キャパシティサマリー（過去ベロシティ vs 現在投入SP）
- ドラッグ&ドロップでバックログからスプリントにチケットを移動

#### チームカレンダー

- メンバーの休暇・不在日を登録・確認
- キャパシティ計算に反映

#### キャパシティ管理

- メンバーごとの稼働可能日数を設定
- プログレスバーで現在のSP投入量と過去ベロシティを比較表示

### 3.6 DSU（Daily Stand-Up）（/dsu）

#### 通常モード

画面構成:
- **上部**: スプリントサマリー（進捗バー、残日数、チケット消化状況）
- **左側**: スプリントゴール別進捗カード（ドラッグで並べ替え可能）+ バーンダウンチャート
- **右側**: メンバー別ステータス入力フォーム

#### メンバーステータス入力

各メンバーについて以下を入力:
- **出席状態**: 出席 / リモート / 欠席
- **昨日やったこと**
- **今日やること**
- **ブロッカー**（障害事項）

#### DSUモード（全画面）

「DSUモード開始」ボタンをクリックすると、全画面表示に切り替わります。

- 15分カウントダウンタイマー付き
- スプリントゴール一覧とメンバーステータスを大きく表示
- プロジェクターやモニター投影に最適

#### DSU履歴

過去のDSUログを日付順に一覧表示。参加人数も確認できます。

### 3.7 グローバル検索

#### 起動方法

- ヘッダーの検索バーをクリック
- **Cmd+K**（Mac）/ **Ctrl+K**（Windows）ショートカット

#### 検索対象

- チケット（タイトル・説明・コメント）
- 中長期ゴール（タイトル・説明）
- スプリントゴール（タイトル・説明）

#### 検索結果（/search）

カテゴリ別にグルーピングして表示。各結果をクリックすると詳細ページに遷移。

### 3.8 プロフィール（/profile）

- 自分の名前・メールアドレスを編集
- パスワード変更（将来対応予定）

## 4. 管理者向けガイド

### 4.1 ユーザー管理（/users）

管理者（ADMIN）が利用可能です。

#### ユーザー一覧

- 全ユーザーをテーブル形式で表示
- メールアドレス、名前、ロール、作成日を確認

#### ユーザー招待・作成

1. 「ユーザー追加」ボタンをクリック
2. メールアドレス、名前、初期ロールを入力
3. 作成後、メールでの招待通知は手動対応（Phase 1）

#### ロール変更

1. ユーザー一覧でロール列のセレクターをクリック
2. ADMIN / SCRUM_MASTER / PRODUCT_OWNER / DEVELOPER から選択

#### ユーザー削除

ユーザー一覧の削除ボタンから実行。削除は取り消しできないため注意。

### 4.2 チーム設定（/team-settings）

#### チーム基本設定

- **チーム名**: チームの表示名を変更
- **チケットプレフィックス**: チケットIDのプレフィックス（例: "SCR" → SCR-001）
- **ベロシティ計算方式**: ストーリーポイントベース / チケット件数ベース

#### 用語カスタマイズ

チーム固有の用語に変更できます。

| デフォルト用語 | カスタマイズ例 |
|---|---|
| インクリメント | PI、クォーター |
| スプリント | イテレーション |
| 中長期ゴール | PIゴール、クォーターゴール |
| スプリントゴール | ITゴール、イテレーションゴール |

設定画面で各用語のテキストを直接編集し、「保存」をクリックします。

### 4.3 DoDチェックリスト管理

チームレベルのデフォルトDoD（Definition of Done）を定義します。

- チケット作成時にデフォルトDoDが自動的に付与される
- 項目例: 「レビュー済み」「テスト済み」「ドキュメント更新済み」「デプロイ済み」
- 各チケット上で項目の追加・削除・編集も可能

### 4.4 本番環境デプロイ

#### Docker Compose による本番デプロイ

```bash
# .envを本番用に設定
cp .env.example .env
# JWT_SECRET, JWT_REFRESH_SECRET を強力なランダム文字列に変更
# DATABASE_URL を本番DBに設定
# NODE_ENV=production に変更

# 起動
docker compose up -d

# マイグレーション
pnpm db:migrate
```

#### セキュリティ注意事項

- `JWT_SECRET`、`JWT_REFRESH_SECRET` は必ずランダムな長い文字列に変更
- `NODE_ENV=production` を必ず設定（開発用のdev-tokenバイパスが無効化される）
- MinIOの認証情報もデフォルトから変更
- Nginxの設定でHTTPSを有効化（別途SSL証明書が必要）

## 5. データベース管理

### 5.1 データベース構成

PostgreSQL 16 を使用。全テーブルに `organization_id` を持たせ、将来のマルチテナント対応に備えた設計。

#### テーブル一覧

| テーブル名 | 説明 | 主要カラム |
|---|---|---|
| `organizations` | 組織（テナント） | name, slug |
| `users` | ユーザー | email, name, role, organization_id |
| `teams` | チーム | name, ticket_prefix, velocity_mode |
| `team_members` | チーム所属（中間テーブル） | team_id, user_id |
| `increments` | インクリメント（PI期間） | name, start_date, end_date, team_id |
| `long_term_goals` | 中長期ゴール | title, priority, status, commitment, increment_id |
| `sprints` | スプリント | name, start_date, end_date, status, velocity, increment_id |
| `sprint_goals` | スプリントゴール | title, status, long_term_goal_id, sprint_id |
| `tickets` | チケット | ticket_number, title, type, status, priority, story_points, sprint_id, sprint_goal_id |
| `ticket_status_history` | ステータス変更履歴 | ticket_id, from_status, to_status, changed_by_id |
| `ticket_comments` | チケットコメント | content, author_id, ticket_id |
| `ticket_labels` | ラベル定義 | name, color, team_id |
| `dod_check_items` | DoDチェック項目定義 | title, team_id, order_index |
| `dod_check_results` | チケットごとのDoD結果 | ticket_id, dod_item_id, checked |
| `dsu_logs` | DSUログ | sprint_id, date |
| `dsu_member_statuses` | メンバーDSU入力 | dsu_log_id, user_id, yesterday, today, blockers, status |
| `team_calendars` | チームカレンダー（休暇等） | team_id, user_id, date, reason |
| `member_capacities` | キャパシティ | sprint_id, user_id, available_days |
| `system_settings` | システム設定 | organization_id, key, value |

#### ER図の主要リレーション

```
Organization ─┬── User ──── TeamMember ──── Team
              ├── Increment ──── LongTermGoal
              ├── Sprint ──┬── SprintGoal ──── Ticket
              │            ├── DsuLog ──── DsuMemberStatus
              │            └── MemberCapacity
              └── SystemSetting
```

### 5.2 マイグレーション操作

Prisma を使用したスキーマ管理です。

```bash
# マイグレーション作成・適用（開発時）
pnpm db:migrate

# Prismaクライアントの再生成（スキーマ変更後）
pnpm db:generate

# シードデータ投入
pnpm db:seed

# Prisma Studio（ブラウザでDBを閲覧・編集するGUIツール）
pnpm db:studio
```

### 5.3 Prisma Studio の使い方

```bash
pnpm db:studio
```

ブラウザで http://localhost:5555 が開きます。全テーブルのデータを閲覧・編集・追加・削除できるGUIツールです。

- テーブル一覧から対象を選択
- レコードをクリックして直接編集
- 「Add record」で新規レコード追加
- フィルタ・ソート機能あり

### 5.4 直接接続（psql）

Docker環境の場合:

```bash
# コンテナ内のpsqlに接続
docker compose exec db psql -U scrum_user -d scrum_db

# ローカルのpsqlから接続
psql postgresql://scrum_user:scrum_pass@localhost:5432/scrum_db
```

よく使うクエリ例:

```sql
-- ユーザー一覧
SELECT id, email, name, role FROM users;

-- アクティブスプリントのチケット状況
SELECT t.ticket_number, t.title, t.status, t.story_points, u.name as assignee
FROM tickets t
LEFT JOIN users u ON t.assignee_id = u.id
JOIN sprints s ON t.sprint_id = s.id
WHERE s.status = 'ACTIVE'
ORDER BY t.priority;

-- スプリントごとのベロシティ
SELECT s.name, s.velocity,
  COUNT(t.id) as total_tickets,
  SUM(CASE WHEN t.status = 'DONE' THEN t.story_points ELSE 0 END) as done_points
FROM sprints s
LEFT JOIN tickets t ON t.sprint_id = s.id
WHERE s.status = 'COMPLETED'
GROUP BY s.id, s.name, s.velocity
ORDER BY s.end_date;

-- 中長期ゴールの達成率
SELECT ltg.title, ltg.status,
  COUNT(sg.id) as total_sprint_goals,
  COUNT(CASE WHEN sg.status = 'ACHIEVED' THEN 1 END) as achieved
FROM long_term_goals ltg
LEFT JOIN sprint_goals sg ON sg.long_term_goal_id = ltg.id
GROUP BY ltg.id, ltg.title, ltg.status;
```

### 5.5 バックアップとリストア

```bash
# バックアップ（Docker環境）
docker compose exec db pg_dump -U scrum_user scrum_db > backup_$(date +%Y%m%d).sql

# リストア
docker compose exec -T db psql -U scrum_user scrum_db < backup_20260413.sql
```

### 5.6 DB不要モード（モックルーター）

`DATABASE_URL` 環境変数が未設定かつ `NODE_ENV` が `production` 以外の場合、バックエンドは自動的にモックルーターを有効化します。

- `apps/backend/src/dev/mockRouter.ts` にハードコードされたモックデータを返す
- UIのレイアウト確認やフロントエンド単体の開発に利用
- 認証は `dev-token` によるバイパスで動作

## 6. システム構成・アーキテクチャ

### 6.1 技術スタック

| レイヤー | 技術 |
|---|---|
| フロントエンド | React 18 + TypeScript + Vite |
| UIコンポーネント | shadcn/ui + Tailwind CSS |
| 状態管理 | TanStack Query（サーバー状態） + Zustand（クライアント状態） |
| ドラッグ&ドロップ | dnd-kit |
| グラフ | Recharts |
| バックエンド | Node.js + TypeScript + Hono |
| ORM | Prisma |

## 7. トラブルシューティング

### ログイン画面が表示されない

- フロントエンドが起動しているか確認: `pnpm dev:frontend`
- ブラウザで http://localhost:3000 にアクセス
- コンソールにエラーが出ていないか確認

### APIエラー（401 Unauthorized）

- JWTトークンが期限切れの可能性 → 再ログイン
- 開発環境: 「Dev Login」ボタンを使用
- `JWT_SECRET` が未設定の場合、本番認証は機能しません

### API接続エラー（Network Error）

- バックエンドが起動しているか確認: `pnpm dev:backend`
- ポート4000が他のプロセスに使用されていないか確認
- `DATABASE_URL` 未設定ならモックモードで動作（正常動作）

### DBマイグレーションエラー

```bash
# Prismaクライアントの再生成
pnpm db:generate

# マイグレーションのリセット（開発環境のみ。データが全削除される）
cd apps/backend && npx prisma migrate reset
```

### Docker Compose 起動失敗

```bash
# ログ確認
docker compose logs -f

# 全コンテナ停止・削除してやり直し
docker compose down -v
docker compose up -d
```

### Vite ビルドエラー

```bash
# フロントエンドディレクトリで実行
cd apps/frontend && npx vite build

# TypeScriptエラーの確認
npx tsc --noEmit --project apps/frontend/tsconfig.json
npx tsc --noEmit --project apps/backend/tsconfig.json
```

### ポート競合

| サービス | デフォルトポート |

| フロントエンド | 3000 |

競合する場合は `.env` やコマンドラインでポートを変更してください。
